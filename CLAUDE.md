# chekipedia

地下アイドルの特典会レギュレーションを横断的に検索・閲覧できるまとめサイト。

**このプロジェクトは Claude Code による AI 駆動開発を前提としています。**

## 課題とユーザー

**課題**：地下アイドルの特典会レギュレーションは、公式情報を追えば見つかることもあるが、手間がかかる。見つからないこともある。調べる労力を減らし、横断的に確認できるようにすることが目的。

**閲覧ユーザー**：地下アイドルファン全般。

**投稿ユーザー**：当面は管理者（開発者本人）のみ。将来的には善意の一般 Twitter ユーザが自発的に追加してくれることを期待するWikipedia的モデル。グループ運営もその一般ユーザーの一人に過ぎず、特別扱いはしない。

## プロジェクトの背景と制約

### なぜこの技術スタックか

個人開発のため、**ランニングコストの最小化**が最重要制約。月 5,000 円以下が目標で、スモールスタート時はほぼ無料枠内を想定している。この制約から、サーバーレス・従量課金のアーキテクチャ（Lambda, DynamoDB, Cognito）を選んでいる。

フロントエンドを静的書き出しにしているのも同じ理由。アクセスが増えて SEO 強化が必要になれば App Runner（SSR）への移行を検討するが、今はコストより優先度が低い。

### なぜ DynamoDB か

RDB の経験はあるが NoSQL の経験値を積みたいという学習目的も含む。設計の難易度は高いが、アクセスパターンの設計から始める方針。

### シークレット管理

**APIキー・認証情報・パスワード等のシークレットは絶対にリポジトリに入れない。** すべて SSM Parameter Store SecureString で管理する。誤って含めそうになった場合は即座に止めてユーザーに確認すること。

### なぜ Terraform か

使い慣れている点に加え、`terraform plan` で変更の diff が明示的に見えること、将来 AWS 以外のリソース（DNS 等）を管理したくなったときも技術スタックが増えないことが理由。**AWS コンソールでの手作業は行わない**。コンソール操作は再現できず、レビューもできないため。

### 認証について

Cognito + Twitter OAuth を選んでいるのは、地下アイドルファンの Twitter 利用率が高いという理由に加え、OAuth の実装経験を積む目的もある。

## 開発の優先順位

「まず動くものを作る」を優先する。過度な抽象化・将来対応・完璧な設計は後回し。Phase 1（閲覧・管理者入力）を最短で動かすことが最初のゴール。

設計の未確定事項に直面したら、GitHub Issues を確認してから判断すること。
